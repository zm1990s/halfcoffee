# 20251221

音乐响起，碎碎念继续，为什么用 CD 机听古典就如此有感觉。

上次似乎写了一半没写完，这周整体事情不多，时间都浪费在测试新产品上面，测试新东西时总是会有大量问题，查文档，查 Slack，反馈虽然可能最终能定位问题，但总感觉有点浪费时间。

周六一个懒觉起来，随便吃了吃又又又吃坏了肚子，难受，风很大，天气挺冷，在新房子也不知道干什么，定了局改，量了量尺寸就早早去吃饭，吃了混搭的面，吃了面回家又整了些桂圆，随便吃了吃感觉又是吃多了，恰了医生开的药，使劲玩游戏到 1 点，然后睡个对时到今天 12 点，两杯咖啡下肚，主打一个纯粹。

本来其实可以不玩游戏的，就因为刷了个朋友圈又开始嫉妒别人的生活，一下子激素就分泌异常，整理了下之前水母的照片，看着拍了很多，实际上好看的没多少...发了两个朋友圈，秀秀存在感（也可能是对嫉妒的挣扎吧）。

然后问 AI 一个问题：”我中了一种毒，一种喜欢上一个人的毒，这个毒叫什么？“，AI 竟然很有诗意的回复了一段打油诗：

它叫 **“心动”**——
学名 *Infatuation virus*，
无药可解，
只能等时间发酵，
或让那个人成为解药。

接着追问了几句，AI 像心理医生告诉我这都是激素在作祟，作祟周期可达 18~30 个月，仔细想想好像有点道理，曾经有点喜欢微博上一个关注的妹子，后面慢慢地不玩微博，现在再去看，其实已经没有什么感觉了，这可能就是度过了这个时期吧。再仔细想想，好像类似的事情也发生过几次，只是当时没有意识，时间长了后就淡忘了。

白天打扫了下家里很脏的地方，挺累的，但有那么一丢丢成就感。然后规划花圃，看书房灯，思考未来可以做的事情（也就是现在想做但提不起兴趣做的事 orz..）

再往前，周四去看了医生，还好没啥大事，本着去都去了的原则开了点药，最后还是用上了。

周五家里蹲，唯一的一天没有去公司，早上太阳☀️很好，边看剧边刷手机，休息的挺好的，下午睡了一觉窝在床上整 PPT，一整就整到天黑，把家里的剩饭清理清理，晚上收拾垃圾，出去运动一下，回来小腿有点疼。

周内又是公司没啥人，挺无聊的。

好久没写公众号了，家里忙的时候这些事情似乎意义都不大了（以及自己生病的时候）。

昨天有个客户有 AI Security Gateway 的想法，看了下思路很像是在 LLM 和 App 之间加一个安全网关，理想很美好，但是总结了下挑战也不好，核心问题还是在于适配，LLM 的适配以及应用的适配。下面记录下具体的内容：

**AI Security Gateway 考量：**

潜在的优点：
- 通用，介于模型和应用之间，类似于传统的防火墙接入网络中，对环境的改造可预期。

潜在的挑战：
- SSL 卸载：如果需要查看到请求内容，需要做正向 Proxy 代理（App->GW->LLM）
- Schema 兼容性：不同的大模型可能使用不同的 HTTP Body 定义，Gateway 需要能够兼容这些不同的格式（也就是说涉及到不同 LLM 的适配，如果有新的 LLM 也需要进行追踪，是个长久的工作）
- 流式兼容：如果应用要使用流式输出，那中间的检测也需要兼容这种流式输出，同时可能需要分段拼接进行检测，如果检测到有害内容，可能需要撤回原来的数据。这种撤回还需要应用侧进行适配。

上面挑战目前 Dify 平台内都已经有解决方案：
1. 无需 SSL 卸载，通过 Dify 的接口直接可以看到用户输入和大模型输出
2. Dify 自身的模型供应商已经针对不同的 LLM 做好了适配，而且有社区持续维护更新
3. Dify 的内容审查接口支持流式，接口会将输入/输出内容以 100 个字节为单位发送给外部进行检查，如果检测出异常 Dify 会撤回之前输出的内容

潜在实现思路：
- 通过 ebpf 在底层获取交互数据并送检，可解决 SSL 卸载的问题，但是其他问题依然需要考虑
- 基于 Kong 等 API Gateway 开发，同样可解决 SSL 卸载的问题，但是其他问题依然需要考虑
- 基于 OneAPI 等 LLM API Gateway 开发，可解决前两个问题，最后一个依然需要考虑
